Human-in-the-loop

Control how much the operator is involved. The plan-approval mode decides whether the orchestrator must get your sign-off before working; interjection lets you steer mid-run.

Kali offensive-tool server

SPAIDER drives real pentest tools through a Kali container running the kali_server/ MCP server. Enable it and point it at your Kali host; its tools (nmap, sqlmap, hydra…) are assigned to the recon/web_app/network/exploitation/post_exploit agents on session start.

Strips banners/progress/log noise from every offensive tool. Agents can still request the full output per call (raw=true); turning this off sends the complete raw output unchanged.

Outbound proxies

Authenticated HTTP proxy in the form http://user:pass@host:port. The two are independent. Hosts in the no-proxy list connect directly (one per line; e.g. localhost, 127.0.0.1, host.docker.internal).

Client proxy — SPAIDER's own outbound (the LLM API)

Kali proxy — offensive tools in the container

Applied to proxy-aware tools (curl/httpx/gospider/nuclei/wget) via HTTP(S)_PROXY/NO_PROXY; raw-socket tools like nmap can't use an HTTP proxy.

Sub-agent spawning limits

Bound how the agent tree can grow. Framework helper agents (tool_selector / summarizer) don't count toward the per-agent limit.

Parameter presets

Save a model's parameters as a named preset and reuse it on any agent. Apply a preset to an agent below, tweak, and re-save to edit it. Stored in config/model_presets.json.

Per-agent models / endpoints / keys / parameters

All LLM parameters are shown; each agent's provider/model only sends the ones it accepts (e.g. Opus 4.8 ignores sampling params; Haiku 4.5 ignores effort). max_tool_size: cap on tools given to the model (0 = unlimited); over it, a tool_selector agent picks the best subset (mandatory internal tools always kept). param renames: fix endpoint drift, e.g. max_tokens=max_completion_tokens; empty right side (temperature=) drops a parameter. API keys are masked — use 👁 to reveal/copy.

Agents (add / remove)

Built-in agents can't be removed. New custom agents become spawnable by the orchestrator (edit the orchestrator prompt to tell it when to spawn them).

Add custom agent

Agent prompts, skills & per-folder MCP servers

Edit each agent's system prompt; set how it uses each skill — always load (static), may load (the agent loads it on demand via load_skill if a task needs it), or never; edit skill content directly in the skills/ folder. Add MCP servers by pasting a JSON config and naming it (inherited by sub-agents). Toggle, test, or remove each server.

Tool-approval policy

Decide what needs your sign-off. For each tool category, choose auto (runs immediately) or manual (the agent pauses for your approval). This is SPAIDER's main safety control. Active only when Approval mode is manual; run_as_admin is always gated.

Internal tools

Built-in tools available to agents (the Kali offensive tools are listed by the server when connected). Add new local ones in spider/tools/custom.py or spider/tools/pentest.py. Each tool's category drives the approval policy above.

Pricing (USD / 1M tokens)

Per-model token prices drive the cost dashboard. Add a model whose name matches exactly what you set in an agent (e.g. gpt-5.4) so its usage is costed instead of showing $0.

Add model pricing

User accounts

Only administrators manage accounts. Each user sees only the sessions they created; admins see all sessions. Passwords are stored hashed (scrypt) and are never shown. The last administrator cannot be deleted or disabled.

Access roles & grants

Define named access roles and the capabilities each grants, then assign them to users (Users tab). The built-in admin role has every capability and is the only role that can open Settings. Changes here are saved with the Save button.
